Unidad 1: Cuenta en GitHub

En esta unidad se introduce GitHub

Para crear una cuenta en GitHub solo se necesita de un correo

GitHub tiene un rol importante en los equipos de trabajo porque permite unificar los cambios de todo el equipo en un solo lugar, este lugar es el -repositorio-

Un repositorio vendría ser una caja o carpeta en donde se van dejando los archivos de proyecto, y de fácil acceso por todos los miembros de equipo

Para estos repositorios siempre hay un Readme, el cual contiene las instrucciones para instalar el software, saber el estado del proyecto, etc

Un readme contiene: Título, imagen de portada, insignias, índice, descripción del proyecto, estado, demostración de funciones, acceso al proyecto, tecnologías utilizadas, personas contribuyentes, personas desarrolladoras y licencia.

Un commit es para indicar de forma clara y corta el cambio que se hizo con un mensaje en el archivo cambiado/nuevo en sí

Markdown: ¿cómo trabajar con este lenguaje de markup? por Paulo Silveira, link al blog -> https://www.aluracursos.com/blog/como-trabajar-con-markdown

Cómo escribir un README increíble en tu GitHub por Camila Alves, link al blog -> https://www.aluracursos.com/blog/como-escribir-un-readme-increible-en-tu-github

Unidad 2: Commit, VSCode y Equipo

Un commit es lo que indica guardar el estado actual del archivo, útil para el control de versiones

Los commits deben usar imperativos al comienzo y mencionar de forma clara y corta lo que se cambió, creó, eliminó

Se puede acceder a VSCode por medio de GitHub con presionar el punto “.”

Para invitar a compañeros de equipo a trabajar un mismo repositorio se les debe invitar como colaboradores en settings y collaborators, y buscarlos

Para ver el historial de commits se usa git log

Para salir del historial presionar q

Unidad 3: Trabajando Localmente

Para instalar git podemos ver el blog de Bruno Souza **Guía sobre cómo instalar Git en diferentes sistemas operativos**
Se profundiza más con los estados y parámetros en el blog de **Luzdalis Lopez , llamado : **Iniciando un repositorio con Git**

Los parámetros globales que se configuran en Git son: Usuario [user.name] y correo [user.email].

Las buenas prácticas para nombrar archivos son: no usar caracteres especiales, usar guiones en lugar de espacios o guiones bajos, usar minúsculas, evitar nombres largos.

Para evitar más errores se puede ver el blog de **Ingrid Silva** llamado y disponible en : **Buenas practicas en git: evitando errores**

Para descargar un proyecto de GitHub a nuestra máquina local es con git clone
Para actualizar el repositorio local con los cambios del remoto utilizamos git pull

Para agregar todos los archivos modificados o nuevos al área de staging (listos para ser enviados al repositorio remoto) para el próximo commit (la acción de guardar todas las modificaciones) se utiliza “git add . “

Para enviar los cambios locales a GitHub se utiliza git push

Para mostrar los cambios en un archivos antes de hacer commit se usa git diff

Para restaurar un archivo a una versión anterior se utiliza git restore —source [hash] [archivo]

Para ver el estado actual del proyecto se usa git status

Blog disponible: **Nueva exigencia de autenticación por token en Git ¿qué es y qué debo hacer?** gracias a **Camila Alves**

Unidad 4: Ramificaciones y Merge

Para ver las ramas existentes se usa git branch

Para crear y cambiar a una nueva rama se usa git checkout -b [nombre rama nueva]

Para cambiar de rama a otra rama sin crear una nueva se usa git switch [rama]

Para enviar los cambios de una rama específica a GitHub se usa git push origin [nombre rama]

Para fusionar los cambios de una rama a otra rama ej. de desarrollo a main, se usa git merge desarrollo (previamente ubicarse en la rama main con git switch main)

Unidad 5: Profundizando en Git y GitHub

git config --global user.name y git config --global [user.email](http://user.email) para configurar Git globalmente

git init inicializa un nuevo repositorio local

git status ver el estado actual del proyecto

git add . agrega todos los archivos modificados para un commit

git commit -m "mensaje” crea una "fotografía" del estado actual del proyecto

git push envía los cambios locales a GitHub

git pull se bajan las actualizaciones desde el repositorio remoto al local

git checkout -b [nombre-rama] se crea una nueva rama y se cambia a ella

git merge [nombre-rama] se mezcla los cambios de una rama a otra

git restore --source [hash] se regresa a un estado anterior usando un hash

diferencia entre Git y GitHub es que Git es la herramienta de control de versiones local, mientras que GitHub es una plataforma en línea para alojar y colaborar en repositorios
